## Phase G1 — Critical: restore the scoring loop

**Finding C1.** The forensic scoreboard — the project's headline "scores
both teams automatically" feature — is wired to a data contract no
producer fulfills. `forensics/scoreboard/scorer.py` joins three indices
on `campaign_id` + `event_type`, but: `campaign_id` is written by nothing
in the tree; `mitre_tagger.tag_and_emit` (the only writer to
`red-team-events-*`) emits neither field; the `campaign_start/end` events
that *do* carry `event_type` (`logger.py`) go to a local file no Logstash
pipeline ingests; `ir-events-*` (MTTA) is written by nothing; and the
`manifest.sha256` the evidence bonus keys on is produced by nothing.
Net effect: `_pairs()` is always empty → red and blue both score 0 →
winner is always a 0–0 "tie", and every real Suricata alert is counted as
a *false positive* (deducting points) because none carry `campaign_id`.

| # | Item | Size | Acceptance |
|---|---|---|---|
| G1a | **Mint a per-run `campaign_id` and thread it through emission.** `runner.py` generates a UUID per invocation (per single campaign, and per stage of `full-killchain`); passes it to `MitreTagger`. `tag_and_emit` adds `campaign_id` and `event_type` (`campaign_start` before `run()`, `campaign_end` after) to every doc it posts to `red-team-events-*`. | S | A `full-killchain` run produces `red-team-events-*` docs where every doc has a non-empty `campaign_id`; each stage has exactly one `campaign_start` and one `campaign_end`. Unit test mocks `requests.post` and asserts both fields present in the payload. |
| G1b | **Stamp `campaign_id` onto the matching detection events** so the MTTD join resolves. Either (preferred) Logstash enriches Suricata alerts from a shared correlation key the campaign sets (e.g. a marker in the attack traffic / a side-channel doc), or the tagger writes a correlation doc the scorer joins on. Document the chosen mechanism in `scorer.py`'s `_pairs()` docstring so it stops lying about a join that can't happen. | M | After a campaign + its detection fire, `scorer._pairs()` returns ≥1 row with a non-`None` `alert_ts`. |
| G1c | **`playbook_engine.py` emits `ir-events-*`** with `event_type:playbook_complete` + the `campaign_id` it was invoked for, on successful playbook completion (MTTA signal). | S | Running a playbook writes one `ir-events-*` doc; unit test asserts the emitted shape; `scorer` response score becomes non-zero in the G1e integration test. |
| G1d | **Unify the evidence manifest filename.** Pick `manifest.sha256` (what `scorer._evidence_bonus` expects) and make `chain_of_custody.py` / `collect_evidence.py` write that name, OR change the scorer to read the existing `custody.json`/`manifest.json`. One name, one writer, one reader. | XS | After `collect_evidence` runs, the scorer's evidence bonus is > 0; a test asserts the produced filename matches what the scorer scans for. |
| G1e | **Integration assertion that scoring actually works.** Extend `tests/integration/test_killchain.py`: after `full-killchain` + at least one IR playbook, `compute_final_scores()` returns **non-zero red AND blue totals**, `false_positives == 0` for matched alerts, and `winner != "tie"` (unless genuinely tied). This is the test that would have caught C1. | S | `pytest -m integration` (or the unittest discover path) fails on `main` today and passes after G1a–G1d. |

## Phase G2 — High: detection/response credibility + CI truthfulness

| # | Item | Size | Acceptance |

| G2a | **(H1) Quote the Elasticsearch healthcheck URL.** `docker-compose.yml:274` — the unquoted `&` backgrounds curl and makes the probe a no-op `timeout=5s` assignment that always exits 0, so ES reports healthy even when down. Wrap the URL in single quotes. **Do this first** — it's a one-liner and it currently masks the exact ES-readiness failures that will confound G1 debugging. | XS | `docker compose config` still valid; killing ES makes `docker compose ps` report the container `unhealthy` within the interval. | G2b | **(H2) Make the Sigma rules real, or descope them.** Today nothing loads the compiled EQL into Kibana's detection engine, and the `webserver`/`proxy`/`auth`/`file_event` logsources have no ingest path (only the MITM syslog path reaches ES). No middle state that advertises 7 detections when 1 can match. | L | Each shipped Sigma rule has a live ingest path; a test maps every rule's `logsource` to a Logstash input. README detection inventory matches reality. |
| G2c | **(H3) Make `block_ip.sh` enforce where traffic flows, or relabel it.** It currently runs `iptables` inside the blue-team netns, which attacker→victim traffic never transits — so the "block" is a no-op the playbooks log as success. Re-implement at the docker-network layer (like `isolate_host.sh`), or explicitly relabel the playbook step as a tabletop drill. | S | A blocked IP can no longer reach the victim after the action runs (integration check), OR the playbook step + docs state it is simulated. |

## Sequencing recommendation

1. **G2a first** (one-liner; unmasks ES failures that would otherwise
   confound G1 debugging).
2. **G1a → G1b → G1c → G1d → G1e** in order — this is the critical path;
   the feature is non-functional until G1e passes.
3. **G2b/G2c** next — restores blue-team credibility now that scoring is
   live and can actually reflect detections.
4. **G3** as a single "correctness + honesty" sweep (profile breakage,
   security default, doc accuracy).
5. **G4** as drive-bys when touching the relevant area, plus the G4e
   hygiene batch in one PR.
